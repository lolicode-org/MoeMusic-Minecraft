# MoeMusic Platform Architecture and Maintenance Notes

This document collects architecture, build configuration, and maintenance guidelines for the MoeMusic platform implementation.

It's strongly advised to read it before making design decisions or starting work, and update it when making durable decisions that should be remembered.

> [!IMPORTANT]
> This document may not be accurate. Most content are generated by AI based on the internal changelog when this mod was initially developed. There might be workarounds, dirty hacks, and version-specific quirks that are not fully captured here. Some old toolchains and build scripts are awkward and what I can do is to make them *just* work and produce a working jar, but they might not be the *right* way to do things. **Don't** take any of the content here for other unrelated projects.

## Scope

The platform project adapts the version-independent MoeMusic shared kernel (`:api`, `:core`, and `:client-core`) to Minecraft loaders and runtime environments. Unlike the shared project, the platform implementation contains Minecraft and loader-specific dependencies. 

> [!IMPORTANT]
> The platform implementation does not expose or guarantee any public API for plugins. It serves purely as an adapter layer between the shared MoeMusic core and Minecraft.

The project is structured into three main modules:
- **`:platform-common`**: A source-only directory containing loader-neutral Minecraft-facing adapters. This includes commands, client UI, client HUD, network packet adapters, and audio output integration. It is not compiled as a standalone JAR; instead, loader modules include its source files directly.
- **`:fabric`**: The Fabric-specific bootstrap, lifecycle wiring, Fabric Permissions API bridge, Mod Menu integration, and Fabric Cloth Config panel.
- **`:neoforge`** (and legacy **`:forge`** on older branches): The Forge/NeoForge-specific bootstrap, lifecycle wiring, LuckPerms adapter, and loader-specific Cloth Config integration.

## Build System and Packaging

### Compilation and Shading
- **Source-Only Common Module**: `:platform-common` is not a Gradle subproject or runtime dependency. Loader modules add its source directories directly to their source sets, generate `MoeMusicPlatformBuildInfo`, and compile its classes directly into the loader JAR.
- **Dependency Shading**: Platform builds use `ShadowJar` to bundle shared-kernel runtime dependencies (such as Wire, ktoml, and LavaPlayer transitives), along with the `:client-core` artifact (which transitively bundles `:core` and `:api`).
- **JPMS and Package Splitting**: Forge/NeoForge launches through a JPMS module layer. To prevent split-package errors, keep loader-specific code out of `platform-common` package namespaces (e.g., place Forge-specific permission bridges in `org.lolicode.moemusic.forge.*` rather than `org.lolicode.moemusic.platform.*`).

### Dependency Resolution Modes
The platform build supports two dependency modes:
1. **Composite Build**: Development builds locate `../shared` and dynamically include it via `includeBuild("../shared")`.
2. **Published Artifacts**: If the sibling `../shared` directory is absent, the build falls back to pulling published artifacts from maven packages (e.g., Codeberg or GitHub Packages) under the `org.lolicode.moemusic` group.

## Bootstrap and Lifecycle

- **Thin Loader Bootstrap**: Keep loader-specific client bootstrap thin. For example, in Fabric, it should only handle loader/event integration (`ClientLifecycleEvents`, `ClientPlayConnectionEvents`, `ClientTickEvents`, `KeyMappingHelper`, HUD registration, and `FabricLoader` lookups). Non-loader client bootstrap (keybinding definitions, hotkey handling, join-tip UI, and config/audio runtime startup) belongs in `:platform-common`.
- **LavaPlayer Native Bootstrap**: LavaPlayer native library extraction path selection is managed in `core.audio.LavaPlayerNativeBootstrap`, but the platform/bootstrap code must pass Minecraft config and game directories before any LavaPlayer manager instance is initialized.

## Multi-Version Maintenance

MoeMusic targets multiple Minecraft versions using Git branches (e.g., `version/26.1`). Maintenance of these branches requires attention to version-specific constraints:

### Java & Mixin Versions
- Default branch (Minecraft 26.1.x) targets **Java 25**.
- Older branches target **Java 17** or **Java 21**. Ensure that the Mixin compatibility level in version-specific mixin configs is correctly aligned (e.g., `compatibilityLevel = JAVA_21` on 1.21.1, `JAVA_17` on 1.20.1).

### Packaging Pitfalls (Pre-26.1 Branches)
- On pre-26.1 branches, public mod JARs must be assembled *inside each loader module* before the loader's remap/reobf step. Do not build release JARs at the platform root by shadowing a loader subproject's raw JAR; doing so produces JARs containing development mappings, causing `NoClassDefFoundError` crashes in production.

### Forge JPMS Loader Scan (1.21.1-)
- On Minecraft 1.21.1-, ForgeGradle 7 scans `build/resources/main` as the mod file. All compiled Java/Kotlin classes (including shared `:platform-common` sources and Java mixins) must be compiled into `build/resources/main` so that FML can locate the mod. Do not duplicate class outputs from `build/classes/java/main` to avoid JPMS split packages.

### Minecraft 1.19- Render Compatibility
- Minecraft 1.19- client screens/HUD render through `PoseStack` instead of `GuiGraphics`. A compatibility wrapper `platform.client.ui.GuiGraphics` is maintained on that branch to keep common UI code unified with newer versions.

## Package Ownership

The platform adapters and loaders are organized by feature packages in `platform-common` (or loader modules for loader-specific features):

- **`org.lolicode.moemusic.platform.runtime`**: The Minecraft composition root and bootstrap initialization logic.
- **`org.lolicode.moemusic.platform.player`**: Minecraft player session mapping and adapter implementations.
- **`org.lolicode.moemusic.platform.network`**: Minecraft network layer, mapping Bad Packets events to `ServerPacketHandlers`.
- **`org.lolicode.moemusic.platform.command`**: In-game player and admin chat command handlers.
- **`org.lolicode.moemusic.platform.text`**: Shared utility layers for version-agnostic text component creation.
- **`org.lolicode.moemusic.platform.chat`**: Platform-specific chat message sending and layout adaptations.
- **`org.lolicode.moemusic.platform.permission`**: Core permission bridges and checks. Loader-specific implementations live in `:fabric` and `:neoforge` under this package.
- **`org.lolicode.moemusic.platform.client.audio`**: Client-side audio sink implementation binding to standard OpenAL outputs.
- **`org.lolicode.moemusic.platform.client.bootstrap`**: Client keybinding registrations and key mapping setups.
- **`org.lolicode.moemusic.platform.client.i18n`**: Client-side localization resolver fallback to Minecraft language resources.
- **`org.lolicode.moemusic.platform.client.network`**: Client network adapters sending requests to the logical server.
- **`org.lolicode.moemusic.platform.client.playback`**: Sync handlers for client playback state updates.
- **`org.lolicode.moemusic.platform.client.ui`**: Game HUD screens, player GUI overlays, and client notifications.

## Core Platform Subsystems

### 1. Networking (Bad Packets)
- Networking is adapted to Minecraft using the Bad Packets library for platform-neutral packet routing.
- Player join/leave session transitions are captured by the loader modules and bridged directly into `core.network.ServerPacketHandlers` to manage active sessions.

### 2. Audio Integration
- Client audio player orchestration resides in `:client-core` (`ClientAudioPlayer`).
- The platform adapter implements the hardware-facing sink `OpenAlAudioOutput` to bind to the client's audio device.

### 3. Text Representation & Localization
- **Minecraft Text Wrapper**: All Minecraft-facing text creation must utilize `org.lolicode.moemusic.platform.text.McText.literal(...)` or `McText.translatable(...)` instead of raw Minecraft `Component` constructor calls. This maintains cross-version compatibility.
- **Client Translation Resolution**: Translation of plugin-provided `LocalizedText` must go through `platform.client.i18n.ClientLocalization`. Standard Minecraft translations (`Component.translatable(...)`) only search vanilla resource packs, which misses standalone plugins' JSON translation keys. `ClientLocalization` queries the core `Localization` register first before falling back to Minecraft's language manager.

### 4. Client Toasts and UI
- To display multiline system toasts correctly, avoid using raw `SystemToast.addOrUpdate(...)` with newline characters (`\n`), which does not wrap in the Minecraft HUD. Use `SystemToast.multiline(...)` or the helper `org.lolicode.moemusic.platform.client.ui.ClientNotificationsKt.showWrappedSystemToast`.

### 5. Permissions
- Advanced permissions are delegated to loader-local modules:
  - Fabric binds through `fabric-permissions-api`.
  - NeoForge handles checks by looking up LuckPerms directly if available, falling back to vanilla OP levels if not.
  - `:platform-common` accesses these checks uniformly via the `AdvancedPermissionChecker` abstraction.
- All permission nodes live in `core/.../permission/PermissionNodes.kt`. `:platform-common` references them from there directly, avoiding duplicate definitions.
